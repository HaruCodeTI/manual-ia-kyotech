## UC01 — Consultar Base de Conhecimento

| Campo | Descricao |
|-------|-----------|
| **Ator primario** | Tecnico de Campo |
| **Descricao** | O tecnico faz uma pergunta em portugues sobre manuais tecnicos Fujifilm e recebe uma resposta com citacoes rastreavels. |

### Pre-condicoes

1. O tecnico esta autenticado no sistema (UC06)
2. Pelo menos um documento foi ingerido na base de conhecimento (UC03)

### Fluxo Principal

1. O tecnico acessa a interface de chat
2. O tecnico digita uma pergunta em portugues (ex: "Como trocar o rolo de pressao da Frontier 780?")
3. O tecnico pode opcionalmente selecionar um filtro de equipamento
4. O frontend envia `POST /chat/ask` com `{question, equipment_filter?, session_id?}` e Bearer JWT
5. O backend resolve ou cria uma sessao de chat
6. O backend persiste a mensagem do usuario no banco
7. O backend reescreve a query via gpt-4o-mini: traduz PT→EN, classifica tipo de documento, extrai equipment_hint
8. O backend executa busca hibrida: vetorial (pgvector, cosine) + textual (pg_trgm, trigram) com fusao 70/30
9. O backend gera resposta em portugues via gpt-4o usando os trechos encontrados, com citacoes `[Fonte N]`
10. O backend faz parse das citacoes referenciadas e filtra apenas as efetivamente usadas
11. O backend persiste a resposta do assistente com citacoes e metadados
12. O frontend exibe a resposta com citacoes clicaveis (cada citacao indica arquivo, pagina e equipamento)

### Fluxos Alternativos

**FA01 — Nenhum resultado encontrado:**
- No passo 8, se a busca hibrida nao retorna resultados
- O sistema retorna mensagem padrao: "Nao encontrei informacoes relevantes nos documentos disponiveis."
- Nao chama o gpt-4o

**FA02 — Continuacao de conversa:**
- No passo 4, se `session_id` e fornecido, o sistema usa a sessao existente em vez de criar uma nova

**FA03 — Falha no query rewrite:**
- No passo 7, se o parse JSON da resposta do gpt-4o-mini falhar
- O sistema usa a query original em portugues sem modificacoes

### Pos-condicoes

1. A pergunta e a resposta estao persistidas na sessao de chat
2. O tecnico pode clicar em uma citacao para visualizar o PDF (UC02)
3. O `session_id` e retornado para continuidade da conversa

## UC02 — Visualizar PDF Seguro

| Campo | Descricao |
|-------|-----------|
| **Ator primario** | Tecnico de Campo |
| **Descricao** | O tecnico clica em uma citacao da resposta do chat e visualiza a pagina do manual renderizada como imagem PNG com watermark dinamico. O PDF original nunca e exposto. |

### Pre-condicoes

1. O tecnico esta autenticado no sistema (UC06)
2. O tecnico recebeu uma resposta com citacoes (UC01)
3. A citacao contem `document_version_id` e `page_number` validos

### Fluxo Principal

1. O tecnico clica em uma citacao `[Fonte N]` na resposta do chat
2. O frontend chama `GET /viewer/info/{version_id}` para obter metadados do documento
3. O backend busca informacoes da versao no banco de dados
4. O backend baixa o PDF do Azure Blob Storage (com cache LRU, TTL 5min, max 10 PDFs)
5. O backend conta o total de paginas do PDF
6. O frontend recebe metadados: `{version_id, source_filename, equipment_key, doc_type, published_date, total_pages}`
7. O frontend chama `GET /viewer/page/{version_id}/{page_number}` para renderizar a pagina
8. O backend baixa o PDF do Azure Blob Storage (usa cache se disponivel)
9. O backend renderiza a pagina como PNG (150 DPI) via PyMuPDF
10. O backend aplica watermark diagonal repetido: `"Kyotech | {user_id} | {timestamp UTC}"`
11. O backend retorna a imagem PNG com headers `Cache-Control: private, no-store`
12. O frontend exibe a imagem da pagina com controles de navegacao

### Fluxos Alternativos

**FA01 — Versao nao encontrada:**
- No passo 3, se `version_id` nao existe no banco
- O sistema retorna HTTP 404: "Versao nao encontrada."

**FA02 — Pagina invalida:**
- No passo 9, se `page_number` esta fora do intervalo (< 1 ou > total_pages)
- O sistema retorna HTTP 400 com mensagem indicando o total de paginas

**FA03 — Erro ao acessar Blob Storage:**
- No passo 8, se o download do PDF falhar
- O sistema retorna HTTP 500: "Erro ao acessar documento."

**FA04 — Navegacao entre paginas:**
- O tecnico usa controles de navegacao para avanccar/retroceder paginas
- O frontend chama `GET /viewer/page/{version_id}/{nova_pagina}` para cada pagina

### Pos-condicoes

1. O tecnico visualizou a pagina do manual com watermark de rastreabilidade
2. Nenhum PDF, SAS URL ou storage_path foi exposto ao frontend
3. O cache LRU do backend mantem o PDF em memoria para navegacao rapida entre paginas

## UC03 — Upload de Documento

| Campo | Descricao |
|-------|-----------|
| **Ator primario** | Gerente / Admin |
| **Descricao** | O admin carrega um manual PDF no sistema. O backend executa o pipeline de ingestion de 6 etapas para extrair, indexar e armazenar o documento. |

### Pre-condicoes

1. O admin esta autenticado no sistema com role `Admin` (UC06)
2. O arquivo e um PDF valido com texto extraivel
3. O arquivo tem no maximo 100MB

### Fluxo Principal

1. O admin acessa a interface de upload
2. O admin seleciona um arquivo PDF
3. O admin preenche os metadados: `equipment_key`, `doc_type` (manual/informativo), `published_date`, e opcionalmente `equipment_display_name`
4. O frontend envia `POST /upload/document` como multipart/form-data com Bearer JWT
5. O backend valida role Admin via `require_role("Admin")`
6. O backend valida: extensao `.pdf`, arquivo nao vazio, tamanho <= 100MB, `doc_type` valido
7. **[1/6]** O backend extrai texto de cada pagina via PyMuPDF e calcula SHA-256 do arquivo
8. **[2/6]** O backend busca ou cria o registro do equipamento (`find_or_create_equipment`)
9. **[3/6]** O backend busca ou cria o registro do documento (`find_or_create_document`)
10. O backend verifica se ja existe uma versao com o mesmo SHA-256 (`check_version_exists`)
11. **[4/6]** O backend faz upload do PDF para Azure Blob Storage no path `{equipment_key}/{published_date}/{filename}`
12. **[5/6]** O backend cria registro de versao no banco e faz COMMIT
13. **[6/6]** O backend divide o texto em chunks, gera embeddings via Azure OpenAI e insere chunks com vetores no banco
14. O frontend exibe resultado: `{success, document_id, version_id, total_pages, total_chunks}`

### Fluxos Alternativos

**FA01 — Arquivo invalido:**
- No passo 6, se o arquivo nao e PDF, esta vazio ou excede 100MB
- O sistema retorna HTTP 400 com mensagem descritiva

**FA02 — Duplicata detectada:**
- No passo 10, se uma versao com o mesmo SHA-256 ja existe
- O sistema retorna sucesso com `was_duplicate: true` e mensagem "Documento ja foi ingerido anteriormente."
- As etapas 4/6 a 6/6 sao puladas

**FA03 — PDF sem texto extraivel:**
- No passo 7, se nenhuma pagina tem texto
- O sistema retorna erro: "PDF nao contem texto extraivel. Pode ser um PDF escaneado — necessario OCR (Fase 3)."

**FA04 — Role insuficiente:**
- No passo 5, se o usuario nao e Admin
- O sistema retorna HTTP 403: "Acesso restrito ao perfil Admin."

### Pos-condicoes

1. O PDF esta armazenado no Azure Blob Storage
2. O documento, versao e chunks com embeddings estao registrados no banco
3. O conteudo do documento esta disponivel para consultas RAG (UC01)
4. O documento pode ser visualizado no viewer seguro (UC02)

## UC04 — Gerenciar Sessoes de Chat

| Campo | Descricao |
|-------|-----------|
| **Ator primario** | Tecnico de Campo |
| **Descricao** | O tecnico lista, abre e deleta suas sessoes de chat para gerenciar o historico de conversas. |

### Pre-condicoes

1. O tecnico esta autenticado no sistema (UC06)

### Fluxo Principal

1. O tecnico acessa a interface de sessoes
2. O frontend chama `GET /sessions?limit=50&offset=0` com Bearer JWT
3. O backend retorna a lista de sessoes do usuario (apenas do proprio usuario)
4. O tecnico seleciona uma sessao para visualizar
5. O frontend chama `GET /sessions/{session_id}` com Bearer JWT
6. O backend retorna a sessao com todas as mensagens (perguntas e respostas com citacoes)
7. O frontend exibe o historico da conversa

### Fluxos Alternativos

**FA01 — Criar sessao manualmente:**
- O tecnico pode criar uma nova sessao vazia via `POST /sessions` com titulo opcional
- O backend cria a sessao e retorna `{id: session_id}`

**FA02 — Deletar sessao:**
- O tecnico seleciona uma sessao e clica em deletar
- O frontend chama `DELETE /sessions/{session_id}` com Bearer JWT
- O backend verifica que a sessao pertence ao usuario e a remove
- Se a sessao nao existe ou nao pertence ao usuario: HTTP 404

**FA03 — Sessao nao encontrada:**
- No passo 6, se `session_id` nao existe ou nao pertence ao usuario
- O sistema retorna HTTP 404: "Sessao nao encontrada."

**FA04 — Paginacao:**
- O tecnico pode navegar entre paginas usando `limit` e `offset`

### Pos-condicoes

1. O tecnico visualizou ou gerenciou suas sessoes de chat
2. Sessoes deletadas sao removidas permanentemente

## UC06 — Autenticar no Sistema

| Campo | Descricao |
|-------|-----------|
| **Ator primario** | Tecnico de Campo / Gerente / Admin |
| **Descricao** | O usuario faz login no sistema via Clerk. O JWT resultante e validado no backend para autorizar acesso aos recursos. |

### Pre-condicoes

1. O usuario possui conta cadastrada no Clerk
2. Para role Admin: `public_metadata.role` deve estar configurado como `"Admin"` no Clerk Dashboard

### Fluxo Principal

1. O usuario acessa a aplicacao
2. O middleware `clerkMiddleware()` do Next.js verifica se ha sessao ativa
3. Se nao ha sessao, o usuario e redirecionado para a pagina de login
4. O usuario preenche suas credenciais no componente `<SignIn />` do Clerk (interface em pt-BR)
5. O Clerk autentica o usuario e retorna sessao + JWT assinado (RS256, TTL 60s)
6. O frontend armazena a sessao via Clerk SDK
7. Para chamadas a API, o frontend obtem JWT atualizado via `getToken()` do Clerk
8. O frontend envia requisicoes ao backend com header `Authorization: Bearer <JWT>`
9. O backend extrai o token via `HTTPBearer`
10. O backend busca a chave publica RSA via `PyJWKClient` no endpoint JWKS do Clerk
11. O backend decodifica e valida o JWT com algoritmo RS256
12. O backend extrai `sub` (user_id) e `metadata.role` do payload
13. Se `metadata.role == "Admin"`, o role e `Admin`; caso contrario, default `Technician`
14. O `CurrentUser(id, role)` e injetado como dependency nos endpoints

### Fluxos Alternativos

**FA01 — Token expirado:**
- No passo 11, se o JWT expirou (TTL 60s)
- O backend retorna HTTP 401: "Token expirado."
- O frontend obtem novo token via `getToken()` e tenta novamente

**FA02 — Token invalido:**
- No passo 11, se o JWT e invalido (assinatura, formato, etc.)
- O backend retorna HTTP 401: "Token invalido."

**FA03 — JWKS indisponivel:**
- No passo 10, se o endpoint JWKS do Clerk nao esta acessivel
- O backend retorna HTTP 503: "Erro ao validar autenticacao."
- O `PyJWKClient` possui cache interno que pode mitigar falhas temporarias

**FA04 — Modo desenvolvimento:**
- Se `CLERK_JWKS_URL` nao esta configurada
- O backend retorna `CurrentUser(id="dev", role="Admin")` sem validacao

**FA05 — Acesso a endpoint com role insuficiente:**
- Apos autenticacao, se o endpoint requer role especifico (ex: `require_role("Admin")`)
- E o usuario nao e Admin, o backend retorna HTTP 403: "Acesso restrito ao perfil Admin."

### Pos-condicoes

1. O usuario esta autenticado e pode acessar os recursos conforme seu role
2. Cada requisicao subsequente carrega JWT atualizado no header Authorization
3. O backend identifica o usuario e seu role em cada requisicao
